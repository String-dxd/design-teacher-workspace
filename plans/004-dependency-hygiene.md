## Why this matters

Three separate hygiene problems compound each other:

1. **Two lockfiles are committed** — `bun.lock` (current, ~Mar 30) and
   `package-lock.json` (stale, ~Mar 13). The repo is Bun-based (`bunfig.toml`
   sets `[run] bun = true`; README documents Bun). A contributor who runs
   `npm install` gets months-old dependency resolutions.
2. **`"nitro": "latest"`** in `package.json:49` is a floating version that
   currently resolves to `3.0.1-alpha.2` — an **alpha**. Any fresh install can
   silently jump to a different (possibly breaking) version; builds are not
   reproducible from `package.json` alone.
3. **`npm audit` reports 23 vulnerabilities (2 critical, 10 high)**. The
   actionable ones within existing semver ranges: vite is at 7.3.1
   (advisories fixed in ≥ 7.4.0 — dev-server path traversal / file read,
   GHSA-4w7w-66w2-5vf9 and related) and vitest is at 3.2.4 (fixed in ≥ 3.2.6
   — GHSA-5xrq-8626-4rwp, arbitrary file read/execute when `vitest --ui` is
   listening). These are dev-machine exposures, not production ones, but
   contributors run these servers daily.

## Steps

### Step 1: Pin nitro to the installed version

In `package.json`, change `"nitro": "latest"` to `"nitro": "3.0.1-alpha.2"`
(exact pin, no caret — alphas do not follow semver compatibility). First
confirm the installed version: `grep '"version"' node_modules/nitro/package.json`.
If it differs from `3.0.1-alpha.2`, pin to whatever IS installed (the goal is
"reproduce today's working state", not an upgrade).

Run `bun install`.

**Verify**: `grep '"nitro"' package.json` → exact version, no `latest`;
`bun run build` → exit 0.

### Step 2: Patch vite and vitest within their semver ranges

Run:

```
bun update vite vitest
```

**Verify**:
- `grep '"version"' node_modules/vite/package.json` → `7.4.0` or higher
- `grep '"version"' node_modules/vitest/package.json` → `3.2.6` or higher
- `bun run build` → exit 0
- `bun run test` → exit 0 (skip if plan 001 not landed)
- `bun run dev` → open http://127.0.0.1:3000, the home page renders without
  console errors; Ctrl-C cleanly.

If `bun update` does not reach those minimums (range ceiling), bump the
ranges in `package.json` to `"vite": "^7.4.0"` and `"vitest": "^3.2.6"`,
re-run `bun install`, and re-verify.

### Step 3: Delete the stale npm lockfile

```
git rm package-lock.json
```

Add a line to `README.md` in the Prerequisites/Getting Started area:

> This project uses **Bun exclusively** — run `bun install`, never
> `npm install` (an `npm` lockfile will drift from `bun.lock`).

**Verify**: `ls package-lock.json` → No such file;
`git status` → deletion staged, `README.md` modified.

### Step 4: Final verification

**Verify**:
- `bun install` → exit 0 and `git status` shows no unexpected lockfile churn
- `bun run build` → exit 0
- `bun run test` → exit 0 (if plan 001 landed)

Stop and report back (do not improvise) if:

- `bun run build` or the dev-server smoke test fails after `bun update` —
